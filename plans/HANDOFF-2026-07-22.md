## The organizing insight

A five-lane audit found ~45 vetted issues. They share one shape, and it is worth
holding onto because it predicts where the next bug will be:

> **Kunai's failure mode is not crashes. It is silent no-ops.**
> A flag is parsed and never consumed. A setting is persisted and never read. A
> manifest field is declared and never enforced. A command is registered and
> never listed. None of it fails a build, so it survives indefinitely, and the
> user meets a feature that "exists" and does nothing.

That is why the first thing built was a **conformance gate**
(`apps/cli/test/unit/architecture/contract-conformance.test.ts`) that fails when
a declaration has no reader. Extend it as you fix things; it is the ratchet.

## Two corrections to earlier claims in this session

Do not trust these if you read them in older notes:

1. **"There is no autoupdate."** **Wrong.** `src/services/update/BinaryAutoUpdater.ts`
   already implements opt-in, binary-only, snooze-aware auto-apply that returns
   `pending-restart` rather than swapping a live session's binary, installs through
   the transactional path, and is started/stopped from `main.ts`. Plan 024 stage 3
   is rewritten to the three _real_ gaps (see below).
2. **"node_modules is gutted."** **Wrong** — that was Bun's isolated-linker layout.
   The real problem was a corrupt `next` extraction (8 of 23 `.d.ts` files present),
   fixed by re-extracting.

## What to do next, in priority order

### 1. `024` — wire publishing into CI (blocking for release)

The launcher works but **nothing publishes the platform packages yet**.

- `bun run build:npm-platform` emits 8 packages to `apps/cli/dist/npm-platform/`.
- **Publish order is load-bearing**: platform packages must go up _before_
  `@kitsunekode/kunai`, or a fresh install resolves optional deps that do not exist.
- `release:publish-tarball` currently publishes only the main package.
- **The binaries in `dist/bin` predate this session's fixes.** Run
  `bun run build:binaries` before release or you ship a binary without any of it.
- Verify with a real container install that has **no Bun on PATH**.

### 2. `019` remaining half — the stranded playback intent

`play-now` now _appears_ and works on the palette route (⇧N, `/notifications` from
browse). It does **not** work when the inbox is opened from inside another overlay:
that path uses a direct `OPEN_OVERLAY` dispatch with no bridge promise, so the
staged intent has no consumer.

**Do not fix this by adding a second global consumer.** The repo's own memory note
on this seam says: reuse the promise bridge, never add another global. The correct
fix is routing that open through `dispatchPaletteCommand`, which is the same
overlay-routing rework as plan 022 / finding DEBT-01. Current behaviour is _safe_
(intent cleared on open, so nothing plays the wrong title) — just not wired.

### 3. `021` stage 3 leftover — language preferences

`EpisodeListRequest` now carries optional `audioPreference`/`subtitlePreference`
and threads them into `ProviderEpisodeListInput`. The playback caller
(`PlaybackPhase.loadAnimeEpisodeOptions`, ~line 4099) does **not** pass them yet —
the profile context is not available at that call site without more plumbing.
Until it is, AllAnime dub users still see the **sub** episode catalogue
(`resolveAnimeAudioIntent` falls back to sub).

### 4. `024` stage 3 — the three real autoupdate gaps

1. `autoApplyBinaryUpdates` has **no settings row** (toggled only via a shell workflow).
2. `pending-restart` is **never surfaced** — an update lands and the user is not told.
3. `updateChannel` **has no reader**, while `kunai upgrade` is advertised as
   "channel-aware" in the CLI help and both installers.

## Landmines — read before you touch these

- **`mock.module`**: 6 test files still use it, unaudited. Rules: spread the real
  module (`...await import(spec)`), and **never stub a pure helper** — the YouTube
  stub invented a fake `yt:video:` id grammar that made the _real_
  `isYoutubeCollectionCatalogId` return false for real ids.
- **`process.env.TZ` in tests**: not hermetic. The runtime caches the zone for
  `Date`, so restoring the var does not undo it — it broke an unrelated
  date-sensitive suite. `test/unit/app-shell/calendar-timezone.test.ts` runs each
  zone in a **subprocess**; copy that pattern.
- **Re-raising signals**: while your own handler is registered, Node delivers the
  re-raised signal to the handler instead of terminating, so the process survives
  and exits 0, silently losing the status. Remove the listener first (see
  `scripts/npm-launcher.mjs`).
- **`docs/installer-reference/`** is untracked **and must stay that way**. It
  contains leaked third-party proprietary source; this repo is public. Do not
  `git add -A`. Use explicit paths when staging.
- **Never test against live user DBs** — copy to `/tmp` first.

## Honest status of the version bump

What a user gets that they did not have before: **npm install works at all**;
notifications can start playback; update notifications actually update; `--zen`
stops corrupting their config; the calendar is correct in their timezone; YouTube
history is filterable; dismiss dismisses. That is a coherent release.

What is _not_ done: the provider contract is still largely decorative (plan 021),
the shell's input layer still drifts from its own registry (plan 022), and the CLI
still advertises flags and docs that are wrong (plan 023). None of those block a
release; all of them are written up.
