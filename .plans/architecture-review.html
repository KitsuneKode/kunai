Theme A — Cross-cutting truth & propagation

Who owns each concept, and how a change reaches everyone that cares.

① Connectivity seam — one reactive source of truth

Strong

Files: services/network/network-availability.ts, NetworkStatusTracker.ts, config.offlineMode, consumers in ink-shell.tsx:721, workflows.ts:200, library-title-detail.tsx:72, source-selection engine.

Problem: "Can I go online?" is answered by two inputs (intent = offlineMode, reality = tracker) ANDed by a free function recomputed at each call site. The tracker isn't reactive — when a search fails and flips it to offline, nothing re-renders on its own; the header only updates if a nearby dispatch happens to trigger a render. Drop network while idle → stale UI.

Solution: a deep Connectivity module owning intent + reality, exposing isOnline() / snapshot() / subscribe(), reactive. Tracker becomes private. network-observation.ts (built today) feeds it; everyone reads container.connectivity.

Benefits: one source of truth (locality); instant propagation to header + source-fallback + hints (leverage); the interface is the test surface — fake the seam, assert UI/fallback.

Deletion test: remove it → the AND-logic and staleness reappear across ~5 call sites. Earns its keep.

② Offline-launch travels three ways — kill the module mailbox

Strong

Files: app/offline-playback-launch.ts:97 (requestUnifiedOfflinePlayback), SearchPhase.ts (consumePendingOfflinePlaybackLaunch), app-shell/types.ts:319 (BrowseShellResult).

Problem: the same launch datum is pushed through three channels: (1) dispatch(SELECT_TITLE/SELECT_EPISODE) — the real handoff; (2) forceCloseRootContent({type:"offline-playback",launch}); (3) the module-global pendingOfflinePlaybackLaunch. #3 is spooky-action-at-a-distance — a new entry point that forgets to consume it silently breaks.

Solution: selection already lives in StateManager (#1 = truth). The loop only needs an "offline playback requested" signal. Collapse to dispatch + a single typed return; delete the module mailbox.

Benefits: explicit data flow; one place to reason about the handoff; no hidden global to keep in sync across entry points.

Deletion test: delete the mailbox → behavior must be re-expressed as a return value (good). It was carrying state, not concentrating logic.

③ Infra reaches up into the UI

Worth exploring

Problem: the lowest layer depends on the top layer, only to ask "is the Ink shell active, so should I suppress stderr chrome?" Layering inversion; the allowlist hides it rather than fixing it.

Speculative

Theme B — Playback & post-playback spine

Evidence from three exploration passes over PlaybackPhase.ts (4,282 lines), the post-playback flow, and the player-control IPC seams.

⑤ PlaybackPhase god-module

Strong

Files: app/PlaybackPhase.ts — 4,282 lines; execute() spans 590–3556 (~3,966 lines) with two nested while(true) loops (main 841, post-play 2771).

Problem: one function interleaves ~25 responsibilities — episode selection, provider resolve, stream/subtitle prep, prefetch, mpv play, history persist, dead-stream invalidation, control routing, autoplay, recommendations, post-play menu. State is scattered across loop-locals (stream, pendingStart, sessionSoftProviderId, resolvedProviderId), StateManager, and a selection coordinator. No one can hold it in context; edits are risky; the only test surface is end-to-end.

Solution: extract deep sub-modules behind small interfaces — EpisodeResolveStep, StreamPrepStep, PlaybackRunStep, PostPlayLoop, with an explicit per-iteration PlaybackIterationState carrying intent. execute() becomes a thin coordinator.

Benefits: each step independently testable; bugs localize; you (and the model) can reason about one step at a time. This is the highest-leverage deepening in the codebase.

Deletion test: n/a — this is decomposition. The win is locality, not removal. Sequence it carefully (behavior-preserving extractions, test after each).

⑥ "What plays next" — a tested seam the runtime ignores

Strong

Files: app/resolve-next-up.ts (resolveNextUp(), pure, tested), vs. three inline branches in PlaybackPhase.ts: catalog autoplay (2516–2670), playlist advance (2697–2748), recommendation advance (2964–3001). A comment at 2962 references resolveNextUp but doesn't call it.

Problem: the decision "what plays after this episode" is duplicated across three imperative blocks with repeated EOF / autoplay-paused / stop-after-current guards. The pure function that should own it exists and is tested — but production reimplements it inline. Bugs fixed in one branch silently miss the others.

Solution: route all three advance paths through resolveNextUp() → it returns a typed NextUp (episode | playlist | recommendation | none); the loop just executes the decision.

Benefits: one decision module = one test surface; guards written once; online/offline/playlist parity by construction.

Deletion test: delete the three inline branches → logic concentrates into the existing pure function. Textbook deepening.

⑦ Playback intent flows through four overlapping channels

Worth exploring

Files: PlayerControlServiceImpl.ts (consume mailboxes lastAction, pendingStreamSelection, …), SessionState.ts (providerSwitchSeq, pause flags, stream), loop-locals in PlaybackPhase, picker path (subscribePickerRequest + workControl.cancelActive).

Problem: there's no single owner for "what should the next loop iteration do?" A provider switch can touch SET_PROVIDER/providerSwitchSeq, recompute → lastAction, and episodePlaybackSelection persistence — reconciled by the loop. Autoplay/autoskip split between SessionState (UI toggle) and PlaybackSessionState (decision); autoskip is frozen at play() time. Correct-by-convention, hard to reason about.

Solution: define one PlaybackIntent queue/bus as the single channel from UI/mpv → loop; mailboxes + seq counters become its implementation. Pairs naturally with ⑤.

Benefits: one place to see every pending user gesture; deterministic reconciliation; live autoskip toggle becomes possible.

⑧ Post-playback surface drift

Worth exploring

Files: root-content-state.ts:5 (RootContentKind "post-playback"), playback-session-controller.ts (phase "post-playback"), but ink-shell.tsx:1547 mounts kind:"playback" for the post-play menu.

Problem: the type system advertises a "post-playback" surface that is never used; the actual post-play menu masquerades as "playback". Anyone keying transitions/clears off RootContentKind will be misled. Cheap clarity bug.

Solution: either mount the post-play menu as kind:"post-playback" (and key screen-clear/transition logic off it), or delete the unused kind. Pick one and make the type honest.

Benefits: the type reflects reality; transition/clear logic can distinguish play vs post-play correctly.

⑨ Offline continuity half-wired

Worth exploring

Files: OfflineRunwayService.ts:19 (trigger "offline-playback-complete" — defined, never enqueued from playback), OfflineLibraryService.ts:111 (savePlaybackHistory — only used in tests).

Problem: two seams exist for offline-aware post-playback (auto-replenish runway after watching, offline-specific history save) but neither is connected — all history/runway side-effects funnel through the online-shaped PlaybackPhase path. So "watched the last downloaded episode → fetch more" never fires at episode end.

Solution: decide intent: if runway-on-complete is wanted, call enqueueEvaluation(..., "offline-playback-complete") from the unified teardown when the played source was local; otherwise delete the dead trigger + method so the surface stops lying.

Benefits: either real offline continuity, or a smaller honest surface. Both beat dead-but-advertised seams.

Deletion test: if no caller is intended → delete (pure dead weight). If intended → wire it where teardown already runs.

Top recommendation & sequencing

Tackle in this order — each unblocks the next and keeps risk low:

  1. 1.
    ① Connectivity seam — small, self-contained, immediately makes today's network work fully real and reactive. Lowest risk, visible payoff. Start here
  2. 2.
    ② Offline-launch mailbox + ③ infra inversion — two cheap, contained cleanups that remove the hidden side-channels and the layering allowlist entries.
  3. 3.
    ⑥ resolveNextUp() adoption — high leverage, mostly routing existing decisions through an existing tested function. De-risks ⑤.
  4. 4.
    ⑤ PlaybackPhase decomposition + ⑦ PlaybackIntent bus — the big one. Do it last, behavior-preserving, one extraction at a time, tests green after each. ⑥ and ⑦ make this tractable.
  5. ·
    ⑧ surface drift & ⑨ offline continuity — fold in opportunistically while touching post-play; each is a tiny decision (wire or delete).
