# 0.3.0 Review Gate and Post-Release Cleanup

> **For the reviewing agent:** you are a _reviewer_, not an implementer. Do not fix
> what you find unless a section explicitly says so. Produce findings with evidence.
> Use `superpowers:verification-before-completion`: never report a check as passing
> without pasting the command output that proves it.

**Goal:** independently verify the three 0.3.0 implementation tracks actually work —
as code _and_ as user-facing behaviour — then perform a bounded cleanup once 0.3.0
is tagged.

**Why a separate reviewer:** every track is being implemented by the agent that
planned it. Today's session produced two bugs that passing tests did not catch
(an infra→providers boundary violation hidden behind a Turbo cache, and a
release workflow that has been broken since June while every local gate stayed
green). Self-review reproduces those blind spots. This gate exists to break them.

## Global constraints

- Do not use a worktree.
- Read-only on product code during §1–§4. Cleanup in §6 is the only write phase.
- Never run destructive commands against the live user profile. Copy
  `~/.local/share/kunai/kunai-data.sqlite` to a scratch path for any DB inspection
  and delete the copy afterwards.
- Run everything with `bun` / `bun run`.
- Turbo caches test results. Before trusting any suite, run it once with
  `--force` (or delete `.turbo`) so a cached pass cannot mask a real failure.

## 2. The CI bug that hid from every local gate (highest priority)

`.github/actions/setup-bun-monorepo` previously ran `actions/checkout` inside
itself. A _local_ composite action cannot be resolved before the repo is on
disk, so any job whose first step was that action failed before its internal
checkout could run. `release.yml` had no checkout of its own. Evidence: the
composite landed 2026-06-27, the last tag is `v0.2.5` (2026-06-08), and
`apps/cli/package.json` is at 0.2.6 with `.release/kunai-v0.2.6.md` present but
**no `v0.2.6` tag and no published binaries**.

- [ ] Confirm `bun test test/unit/scripts/ci-bootstrap-contract.test.ts` passes
- [ ] Confirm every `uses: ./.github/actions/...` in every workflow is preceded by
      a checkout **in the same job** (the contract test asserts this; verify it by
      reading the workflows too — a test can encode the wrong rule)
- [ ] **Verify on GitHub, not locally:** a real `workflow_dispatch` run of
      `release.yml` (or a dry equivalent) must reach the publish step. A green
      contract test is _not_ evidence the pipeline works.

**This is the single most important item in this document.** 0.3.0 cannot ship
until a real CI run proves it.

## 3. Behavioural review — did the fix fix the user's problem?

For each item: reproduce the _original symptom_ first, then confirm it is gone.
A passing unit test is necessary, not sufficient. Run in a real terminal
(kitty and one plain terminal) against the built binary.

### Track A — shell UX

- [ ] After a search, the highlighted row and Enter act on the _same_ row
- [ ] Esc/`↑`-at-top returns to search without losing the selection
- [ ] Closing details returns to the list, not the search field
- [ ] Bare `m` opens the title-control menu when the list is focused
- [ ] The anywhere-menu chord is **`Shift+M`** — confirm the implementer did not
      ship `Ctrl+Shift+M` or `Ctrl+M`. Both are byte-identical to Enter in legacy
      terminal encoding; a binding for them is silently dead. Reject on sight.
- [ ] Esc during a resolve cancels _and_ returns to the row that was playing
- [ ] Notification rows show posters, not two-letter initials

### Track B — retained PR reliability

- [ ] Overlay retention: opening and closing an overlay over browse does not
      remount browse or lose calendar state
- [ ] No double input delivery for footer-owned letters (press each footer key once,
      confirm one action)
- [ ] Diagnostics panel and exported bundle contain the _same_ evidence
- [ ] Source-inventory quality partition does not resurrect stale streams: play,
      switch quality, replay, confirm the chosen variant persists

### Track C — diagnostics/observability

- [ ] Everything stays **local**: confirm by inspection that no diagnostic or trace
      data leaves the machine on any path except an explicit, user-initiated export
- [ ] Telemetry is **opt-in**: a fresh profile sends nothing before consent. Verify
      with a shadow XDG profile and a network check, not by reading the code
- [ ] The install id is a random salted UUID — **not** derived from hostname, MAC,
      IP, or username. Reject anything hardware-derived
- [ ] The consent prompt states plainly what is sent and how to turn it off
- [ ] A support bundle from a fresh profile is readable, redacted (no absolute home
      paths, no stream URLs with tokens), and small enough to paste into an issue
