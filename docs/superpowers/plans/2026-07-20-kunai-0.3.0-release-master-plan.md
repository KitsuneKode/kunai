# Kunai 0.3.0 Release Master Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver a dependable Kunai 0.3.0 whose installer, release metadata, core playback/continuation features, first-run experience, documentation, and published artifacts agree with reality.

**Architecture:** Implement the approved release spine as seven independently reviewable subsystem plans. Shared ownership contracts land before their consumers: installer safety before lifecycle expansion, queue persistence before shell/playback acknowledgement, runtime behavior before documentation, and verified candidate artifacts before any publication action.

**Tech Stack:** Bun 1.3.x, TypeScript, React 19, Ink 7, SQLite, Bash, PowerShell, Docker, GitHub Actions, Changesets, npm, GitHub Releases.

## Global Constraints

- Release target is exactly `0.3.0`.
- Linux glibc and musl are supported; macOS and Windows are beta; Windows ARM64 is experimental.
- Downloads and offline playback remain beta.
- Protocol registration is Linux-only for 0.3.0.
- Native binary is the primary install path, followed by Bun global, npm global, and source checkout for contributors.
- Native installation must never silently delete package-manager-owned installs.
- Source checkout, runtime data, staging, locks, versions, and launcher paths must remain separate.
- A queue item becomes played only after confirmed `playback-started` evidence.
- Optional timing, subtitle, presence, poster, and opener work must not block core playback or break the shell.
- Publishing npm, creating tags, and promoting a GitHub release require a separate explicit confirmation after every deterministic gate passes.
- Do not use git worktrees in this repository.
- Use `bun`, `bunx`, and `bun run`; never invoke `bun test` directly.
- Stage exact paths only; never use `git add .` or `git add -A`.
- Preserve the unrelated working-tree change in `scripts/generate-release-notes.ts` until its dedicated release-truth task assumes ownership.
- Never stage or commit `docs/installer-reference/claude-code/`.
- Before every commit, run `git status --short` and inspect the task-specific diff.
